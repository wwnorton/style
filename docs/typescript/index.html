General‚Äã

  • No Unused Expressions

We shouldn't have unused expressions in our code. If we have them, we should remove them.

// Bad üëé
a + b;

// Good üëç
let add = a + b;
  • Filename

Name files with camelCase.

Reason: Conventional across many JS teams.

accordion.tsx
myControl.tsx
stringUtils.ts
map.ts

Variables and Function‚Äã

  • Use camelCase for variable and function names

Reason: Conventional JavaScript

// Bad üëé
var FooVar;
function BarFunc() { }

// Good üëç
var fooVar;
function barFunc() { }
  • No Unused Variables

@typescript-eslint/no-unused-vars

Likewise, if we have unused variables, then we should remove them.

// Bad üëé
var num1 = 10;
var num2 = 7;

var result = num1 + 7;

// Good üëç
var num1 = 10;

var result = num1 + 7;

// Good üëç
var num1 = 10;
var num2 = 7;

var result = num1 + num2;
  • Don‚Äôt Use Variables Before they‚Äôre Defined

With variables that are declared with¬†var we can reference the variable before they‚Äôre defined, but the value will be¬†undefined. This is because the variable is hoisted. let and¬†const solve this problem since they aren‚Äôt hoisted. Therefore, we should use¬†let or¬†const variables. This way, if we reference those variables, we‚Äôll get an error.

// Bad üëé
var hoisted;
console.log(hoisted); // will be undefined;

// Good üëç
let notHoisted;
const alsoNotHoisted;

console.log(notHoisted); // Will throw an error (which is what we want in this case);
console.log(alsoNotHoisted); // Will also throw an error (which is what we want in this case);
  • Replace Magic Numbers with Named Constants

@typescript-eslint/prefer-as-const

If we have numbers that are used as constants repeatedly but aren‚Äôt assigned to a constant, then we should assign it to one. This way, we can change it once and the value will be reflected everywhere. Also, named constants tell us the meaning of the number. Therefore, instead of writing:

// Bad üëé
enum Foo { bar = 1 }

// Good üëç
const NUMBER_OF_BARS = 1;
enum Foo { bar = NUMBER_OF_BARS }
  • Don‚Äôt Use async if await isn‚Äôt Used Inside the Function

@typescript-eslint/require-await

We should use¬†async¬†functions only if we have to¬†await¬†something inside it.

For instance, if we have something like:

// Bad üëé
const foo = async () => "bar";

// Good üëç
const foo = async () => {
    const url = "some.domain.com";
    let response = await fetch(url, {
      method: 'GET'
  });
}

then we should use a normal function.

  • Return Awaited Values Consistently

We should not have¬†return¬†and¬†await¬†on the same line since the promises may not have resolved yet.

Instead, put them on separate lines. The only exception to this is that we can put¬†return¬†and¬†await¬†on the same line inside a¬†try¬†block to catch errors from another promised-based function

// Bad üëé
async function foo() {
  return await bar();
}

// Good üëç
async function foo() {
  const val = await bar();
  return val;
}

// Good üëç
async function foo() {
  try {
    return await bar();
  } catch (error) {}
}

Array‚Äã

  • Annotate arrays as foos:Foo[] instead of foos:Array<Foo>.

Reasons: Its easier to read. Its used by the TypeScript team. Makes easier to know something is an array as the mind is trained to detect [].

// Bad üëé
foos:Array<Foo>;

// Good üëç 
foos:Foo[];

Strings‚Äã

  • Prefer single quotes (') unless escaping.

Reason: More JavaScript teams do this (e.g. airbnb, standard, npm, node, google/angular, facebook/react). Its easier to type (no shift needed on most keyboards). Prettier team recommends single quotes as well

Double quotes are not without merit: Allows easier copy paste of objects into JSON. Allows people to use other languages to work without changing their quote character. Allows you to use apostrophes e.g. He's not going.. But I'd rather not deviate from where the JS Community is fairly decided.

// bad üëé
var foo = "bad";

// good üëç
var foo = 'bar';
  • When you can't use double quotes, try using back ticks (`).

@typescript-eslint/restrict-template-expressions

Reason: These generally represent the intent of complex enough strings.

// good üëç
var foo = `this foo has a dynamic ${bar}`;
  • Have Consistent Use of Backticks, Double Quotes or Single Quotes

We should use backticks or quotes in a consistent manner for declaring strings. Better yet, we should use backticks since they‚Äôll create template strings, which are more flexible. They allow expressions to be embedded in it.

Class‚Äã

  • Use PascalCase for class names.

Reason: This is actually fairly conventional in standard JavaScript.

// Bad üëé
class fooBoo { }

// Good üëç
class FooBoo { }
  • Use camelCase of class members and methods

Reason: Naturally follows from variable and function naming convention.

// Bad üëé
class Foo {
    BarVar: number;
    BazFunc() { }
}

// Good üëç
class Foo {
    barVar: number;
    bazFunc() { }
}
  • No Useless Constructors

@typescript-eslint/no-useless-constructor

We shouldn‚Äôt have useless constructors.

// Bad üëé
class A {
  constructor () {
  }
}

// Bad üëé
class B extends A {
  constructor (value) {
    super(value);
  }
}

They‚Äôre both redundant so they should be removed.

Interface‚Äã

  • Use PascalCase for name.

Reason: Similar to class

// Bad üëé
interface themeProps {
    color: string;
}

// Good üëç
interface ThemeProps {
    color: string;
}
  • Use camelCase for members.

Reason: Similar to class

// Bad üëé
interface ColorProps {
    FirstColor: string;
    second-color: string;
}

// Good üëç
interface ColorProps {
    firstColor: string;
    secondColor: string;
}
  • Don't prefix with I

Reason: Unconventional. lib.d.ts defines important interfaces without an I (e.g. Window, Document etc).

// Bad üëé
interface IFoo {
}

// Good üëç
interface Foo {
}

Type‚Äã

  • Use PascalCase for name.

Reason: Similar to class

// Bad üëé
type infoBaseProps = 'htmlFor' | 'className' | 'children' | 'id';

// Good üëç
type InfoBaseProps = 'htmlFor' | 'className' | 'children' | 'id';
  • Use camelCase for members.

Reason: Similar to class

// Bad üëé
type Animal = {
  animalrace: string
}

// Good üëç
type Animal = {
  animalRace: string
}

Type vs. Interface‚Äã

  • Use type when you might need a union or intersection:
// Good üëç
type Foo = number | { someProperty: number }
  • Use interface when you want extends or implements e.g
// Good üëç
interface Foo {
  foo: string;
}

interface FooBar extends Foo {
  bar: string;
}

class X implements FooBar {
  foo: string;
  bar: string;
}

Namespace‚Äã

  • Use PascalCase for names

Reason: Convention followed by the TypeScript team. Namespaces are effectively just a class with static members. Class names are PascalCase => Namespace names are PascalCase

// Bad üëé
namespace fooBaz {
}

// Good üëç
namespace FooBaz {
}

Enum

  • Use PascalCase for enum names

Reason: Similar to Class. Is a Type.

// Bad üëé
enum colorOne {
}

// Good üëç
enum ColorOne {
}
  • Use PascalCase for enum member

Reason: Convention followed by TypeScript team i.e. the language creators e.g SyntaxKind.StringLiteral. Also helps with translation (code generation) of other languages into TypeScript

// Bad üëé
enum Color {
    redHue
}

// Good üëç
enum Color {
    RedHue
}

Null vs. Undefined‚Äã

  • Prefer not to use either for explicit unavailability

Reason: these values are commonly used to keep a consistent structure between values. In TypeScript you use types to denote the structure

// Bad üëé
let foo = { x:123, y:undefined };

// Good üëç
let foo:{ x:number, y?:number } = { x:123 };

  • Use undefined in general (do consider returning an object like {valid:boolean,value?:Foo} instead)
// Bad üëé
return null;

// Good üëç
return undefined;

  • Use null where its a part of the API or conventional

Reason: It is conventional in Node.js e.g. error is null for NodeBack style callbacks.

// Bad üëé
cb(undefined)

// Good üëç
cb(null)

  • Use truthy check for objects being null or undefined
// Bad üëé
if (error === null)

// Good üëç
if (error)

  • Use == null / != null (not === / !==) to check for null / undefined on primitives as it works for both null/undefined but not other falsy values (like '',0,false) e.g.
// Bad üëé
if (error !== null) // does not rule out undefined

// Good üëç
if (error != null) // rules out both null and undefined

Tabs vs. Spaces‚Äã

  • Use tabs over spaces.

Reason: Use tabs for indenting your code. With tabs users can choose their desired width. This has positive implications for accessibility and screenreaders, putting people in control of how they want to view the code.

// bad üëé
function foo() {
‚àô‚àôlet name;
}

// bad üëé
function bar() {
‚àôlet name;
}

// good üëç
function baz() {
‚àô let name;
}

Semicolons‚Äã

  • Use semicolons.

Reasons: The usage of semicolons is a core value of our community. Consider the points of the opposition, but be a traditionalist when it comes to abusing error correction mechanisms for cheap syntactic pleasures. Explicit semicolons helps language formatting tools give consistent results. Missing ASI (automatic semicolon insertion) can trip new devs.

// Bad üëé
let x = 5

// Good üëç
let x = 5;

We should put in semicolons in our code instead of letting the JavaScript interpreter add them for us in unexpected places.

  • Remove Useless Semicolons

@typescript-eslint/no-extra-semi

We should remove duplicate semicolons in our code. We only need one semicolon at the end of a statement.

// Bad üëé
let x = 5;;

// Good üëç
let x = 5;
  • Require Semicolons Instead of Automatic Semicolon Insertion

Instead of letting the Javascript interpreter put in semicolons for us, we should put them in ourselves.

// Bad üëé
function returnObject() {
    return //<-----semi colon will be inserted causing an error.
    {
        key: value
    };
}

// Good üëç
function returnObject() {
    return {
        key: value
    };
}

Import / Export‚Äã

  • Avoid export default

Avoid export default in TypeScript because of the maintainability and DX issues outlined in Typescript Deep Dive - Avoid Export Default.

Example: 

// foo.ts
class Foo {
}
export default Foo;
// bar.ts
import Foo from "./foo";
  • If you refactor Foo in foo.ts it will not rename it in bar.ts.
  • If you end up needing to export more stuff from foo.ts (which is what many of your files will have) then you have to juggle the import syntax
// Bad üëé
// foo.ts
export default class Foo {
  return 
  {
    key: value
  };
}

// bar.ts
import Foo from './foo';


// Good üëç
// foo.ts
export class Foo {
    return {
        key: value
    };
}

// bar.ts
import { Foo } from "./foo";
